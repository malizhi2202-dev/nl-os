## 0. 技术栈选定

> 由 2-design 步骤 0 锁定。变栈视为开新 CHANGE（R7.1）。

- **选定**: 自定义 C 语言系统级栈（不在 flow-kit 预定义 8 栈中——这是 OS 级项目）
- **语言**: C11（GCC ≥ 9 / Clang ≥ 14）
- **构建系统**: CMake ≥ 3.16
- **核心依赖**（3 个，vendor 锁定）:
  - ggml（张量运算 + GGUF 加载，vendor 锁定 commit）
  - cJSON v1.7.x（JSON 解析，单文件 vendor）
  - greatest.h（单元测试 header-only）
- **系统依赖**: pthread（POSIX 线程）、Unix domain socket（IPC）、syslog（审计）
- **模型格式**: GGUF v3
- **目标平台**: Linux x86_64 + ARM64（aarch64），glibc ≥ 2.28，内核 ≥ 5.10
- **理由**: C 是唯一能在内核接口、嵌入式设备、高性能推理三个场景同时工作的语言。需求文档明确要求"C 语言适应不同架构、极致优化"。
- **明确排除**:
  - C++：需求明确"C 语言或 C++"，选 C 保持最小依赖面和更简单的交叉编译
  - Rust：生态不成熟于模型推理，且内核集成（eBPF）的 C 兼容层更直接
  - Python/Node.js：需求明确"不依赖 GPU、不装 agent、快速启动"，解释型语言不满足

## 1. 决策清单

| # | 决策 | 备选 | 选择理由 | 取舍代价 |
|---|---|---|---|---|
| **D1** | 四层混合管线（安全网关 → 规则引擎 → 模型推理 → 模板填充） | 纯模型端到端生成 / 纯规则引擎 | 安全网关前置确保危险路径在模型前拦截；规则覆盖 80% 高频场景零延迟；模型只做意图分类不直接生成命令 | 四层管线增加了代码复杂度；Layer 间数据传递需要严格定义接口 |
| **D2** | 权限守护独立进程 + Unix socket IPC | 内嵌到 NL-Shell / D-Bus | 独立进程可被 systemd 管理（崩溃自动重启）；fail-closed 安全边界清晰；Unix socket 比 D-Bus 更轻量无依赖 | 增加 IPC 通信开销（< 1ms per request）；两个进程需协调启动顺序 |
| **D3** | libnlc 共享库（模型 mmap 一次，多进程共享） | 静态链接 / 每进程独立加载 | 树莓派 4 只有 4GB RAM，模型 2GB — 多进程各自加载内存直接爆掉；mmap MAP_SHARED 让内核管理页面共享 | 共享库版本兼容性需要管理；dlopen 增加启动复杂度 |
| **D4** | vendor 外部依赖（锁定 commit） | 系统包管理器 / git submodule | ggml 上游频繁 breaking change；vendor 保证可重复构建；不依赖系统包管理器（嵌入式可能没有） | 需要手动同步上游安全补丁；vendor 目录占用仓库空间 |
| **D5** | fail-closed 安全策略 | fail-open（挂了就放行） | 需求明确"L4 打死不能执行"；权限守护挂了意味着安全边界消失→拒绝所有 NL 操作比放行危险操作更安全 | 权限守护故障 = NL-Shell 基本不可用（但传统命令透传不受影响） |
| **D6** | 规则引擎优先于模型 | 模型优先 | 高频命令需要零延迟+100%确定；规则覆盖的是"确定性高"的场景；模型处理"模糊但安全"的场景 | 规则和模型的边界划分是核心工程挑战——规则太多退化、太少不可靠 |
| **D7** | C11 标准（非 C17/C23） | C17 / C99 | C11 有 `_Generic`/`_Static_assert`/`_Atomic` 等对系统编程有用的特性；旧 ARM 交叉编译链（如 Buildroot 2020）对 C17 支持不完整 | 放弃 C17 的 `[[fallthrough]]`/`<stdatomic.h>` 改进，但影响小 |
| **D8** | `.tools` 先于 `.nl` 交付 | 直接做完整 NL-C | .tools 是 MVP 可验证的最小 NL 编程单元；积累用户数据后再设计 .nl 语法；防止过早抽象 | .tools 的简单性可能限制早期用户的使用场景，但 MVP 目标就是验证核心价值 |
| **D9** | 模型只做意图提取 + 实体识别（分类问题），不做命令生成 | 端到端 seq2seq 生成命令 | 分类问题准确率远高于生成问题（尤其对小模型）；模板填充保证命令结构安全；模型输出的 entity 值通过白名单+stat()验证 | 表达能力受限于预定义的 intent 枚举和模板数量；复杂操作可能需要更多模板 |
| **D10** | Benchmark 先于模型选型 | 先选模型再测试 | 避免"拍脑袋选模型"→ 发现不适用→ 推倒重来；benchmark 是持续投资（每次模型更新都要跑）；数据驱动决策 | MVP 前期需要投入人力建设 benchmark（1000 条标注数据），但这是 C 代码开发前的纯数据工作 |

### 2.2 四层管线数据流

```
NL 输入: "把 downloads 里大于 100M 的视频删掉"
       │
       ▼
┌──────────────────────────────────────────────────────┐
│ Layer 1: 安全网关 (nl_path_safety_check)              │
│                                                      │
│ 输入: raw NL string                                   │
│ 处理:                                                 │
│   1. 剥离 Unicode 控制字符/零宽字符                     │
│   2. 提取 NL 中所有路径关键词 (正则)                   │
│   3. realpath() 规范化所有路径                         │
│   4. 白名单检查: /home /tmp /var /opt /mnt             │
│   5. 黑名单检查: /etc /boot /sys /proc /dev/*          │
│                                                      │
│ 输出: {status: PASS/REJECT, cleaned_input, warnings}  │
│       如果 REJECT → 直接返回用户，不继续               │
└──────────────────────┬───────────────────────────────┘
                       │ PASS
                       ▼
┌──────────────────────────────────────────────────────┐
│ Layer 2: 规则引擎 (nl_rule_match)                     │
│                                                      │
│ 输入: cleaned NL string                               │
│ 处理:                                                 │
│   1. 分词 + 关键词提取                                │
│   2. 规则库匹配 (hash 表查找, O(1))                    │
│   3. 如果命中 → 返回 {template_id, slots}             │
│   4. 如果未命中 → 返回 NULL，进入 Layer 3             │
│                                                      │
│ 规则库示例:                                           │
│   "创建 文件夹 <name>" → {tid:1, slots:{name}}        │
│   "删除 <path> 里 <cond> 的 <type>" → {tid:5,...}    │
│                                                      │
│ 输出: {matched: true/false, template_id, slot_values} │
└──────────────────────┬───────────────────────────────┘
                       │ 未命中
                       ▼
┌──────────────────────────────────────────────────────┐
│ Layer 3: 模型推理 (nl_intent_extract)                  │
│                                                      │
│ 输入: cleaned NL string + few-shot examples (3条)     │
│ 处理:                                                 │
│   1. 构建 prompt (system + examples + user input)      │
│   2. Tokenize → ggml 推理 → 采样                      │
│   3. 解析输出为 JSON → Schema 校验                    │
│   4. 如果 JSON 无效 → 重试 (最多 3 次)                │
│   5. 如果 confidence < 0.6 → 返回低置信度拒绝          │
│                                                      │
│ 输出 JSON:                                            │
│   {                                                   │
│     "intent": "delete",                               │
│     "entities": {                                     │
│       "target": "视频文件",                            │
│       "location": "~/downloads",                      │
│       "filter_type": "size_greater",                  │
│       "filter_value": "100M"                          │
│     },                                                │
│     "confidence": 0.87                                │
│   }                                                   │
└──────────────────────┬───────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────┐
│ Layer 4: 模板填充 (nl_template_fill)                   │
│                                                      │
│ 输入: {intent, entities} (来自 L2 或 L3)              │
│ 处理:                                                 │
│   1. intent → 选择命令模板                             │
│   2. entities → 填充模板槽位                           │
│   3. 路径再次 realpath() + 白名单验证                  │
│   4. 生成最终命令字符串                                │
│                                                      │
│ 模板: "find {location} -name '{patterns}'              │
│         -size +{size} -delete"                         │
│                                                      │
│ 输出: "find ~/downloads -name '*.mp4' -name '*.mkv'   │
│         -size +100M -delete"                           │
└──────────────────────┬───────────────────────────────┘
                       │
                       ▼
                  权限守护审核 → 用户确认(L3) → 执行
```

### 2.3 IPC 协议（NL-Shell ↔ 权限守护）

```
   │  }                                         │
   │                                            │
   │  ◀──── RESPONSE ──────────────────────     │
   │  {                                         │
   │    "decision": "confirm",                   │
   │    "level": "L3",                          │
```

```
   │    (fail-fast)            │
   │                           │
   │  i ≤ N? ─── 是 ───────────┘
   │
   │ 否
   ▼
nl-run exit code = 0
```

## 5. 风险

| # | 风险 | 类型 | 影响 | 概率 | 缓解 |
|---|---|---|---|---|---|
| **R1** | 2B 模型中文 NL 意图分类准确率不足（< 70%） | 实现风险 | NL-Shell 核心功能不可用，用户输入频繁被拒绝 | 中 | 先建 benchmark → 数据驱动选型；备选：0.5B 模型做有限意图集（7 类）准确率更高；规则引擎覆盖 80% 场景兜底 |
| **R2** | ggml 上游 breaking change 导致 vendor 版本无法加载新版 GGUF 模型 | 长期债务 | 社区新模型无法使用，需要手动维护 ggml fork | 中 | 锁定 GGUF v3；提供 `nl-model-convert` 格式转换工具；每个大版本评估一次上游更新 |
| **R3** | 权限守护 IPC 瓶颈——高并发时 Unix socket 队列溢出 | 上线风险 | NL-Shell 操作延迟增加或超时 fail-closed | 低 | MVP 单用户场景不触发；二期加 socket 连接池 + 异步处理；监控队列深度 |
| **R4** | 模型加载 2s + 推理 500ms 导致首次 NL 交互体验差 | 实现风险 | 用户感觉"这个 shell 很慢" | 中 | 规则引擎覆盖 80% 高频命令（零延迟）；模型 mmap 常驻（热缓存 < 100ms）；可选更小模型（0.5B）做意图分类 |
| **R5** | 对抗性 NL 输入绕过安全网关（路径编码/Unicode 混淆） | 安全风险 | L4 黑名单被绕过，危险命令可能执行 | 中 | 安全网关先做 Unicode 规范化（NFKC）；realpath() 解析所有间接路径；对抗性样本加入 benchmark |
| **R6** | C 语言内存管理错误（buffer overflow/use-after-free）导致 NL-Shell crash | 实现风险 | 用户体验差，可能丢失输入历史 | 高 | 严格 code review + 静态分析 (clang-analyzer/coverity)；所有外部输入用长度限定宏；fuzzing (libFuzzer) 针对 NL 输入解析 |
| **R7** | 社区不接受"又一个 shell"——生态冷启动失败 | 上线风险 | 用户量不足，项目停滞 | 中 | 兼容传统命令透传（不是替代 bash，是增强）；聚焦边缘设备/IoT 蓝海场景；开源社区驱动 + 发行版合作 |

## 6. 不在范围

- **eBPF 内核权限拦截**：Phase 2，MVP 用户态已足够
- **模型 OTA 更新**：Phase 2，MVP 使用系统包管理器分发模型
- **动态 few-shot 检索（向量相似度）**：Phase 2，MVP 使用静态 few-shot 或零样本
- **多用户协作/权限隔离**：Phase 3
- **`.nl` 完整编程语言（函数/包/导入/类型）**：Phase 2
- **GUI/语音/多模态输入**：out（永久）
- **Windows/macOS 支持**：out（永久）
- **GPU 加速推理**：out（永久——核心价值是纯 CPU 运行）

## 9. 架构沉淀建议

### 9.1 新增的可复用抽象

| 路径 | 能力 | 触发场景 | 复用建议 |
|---|---|---|---|
| `src/libnlc/path_safety.c` | 路径规范化 + 白名单 + 黑名单校验 | 任何需要验证文件路径安全性的场景 | 其他 Linux 安全工具可独立使用此模块 |
| `src/libnlc/template_fill.c` | 意图+实体 → 命令模板填充引擎 | 任何 NL→结构化输出的场景 | 模板定义格式可复用；二期 NL-C 包系统可基于此扩展 |
| `src/nl-perm/perm_engine.c` | 可配置的 L1-L4 权限判定引擎 | 任何需要操作分级的系统 | 规则文件格式 (`perm_rules.conf`) 可作为 Linux 安全组件的标准接口 |

### 9.2 新增 / 改变的项目级技术决策

| 决策 | 取值 | 影响范围 | 推翻代价 |
|---|---|---|---|
| 权限守护 fail-closed | 守护进程不可达 = 拒绝所有 NL 操作 | NL-Shell + nl-run | 高——改为 fail-open 需要在所有调用点加防御逻辑 |
| 模型只做分类不做生成 | intent 枚举 + 实体提取，模板填充生成命令 | 整个 NL 理解管线 | 高——改为 seq2seq 需要重新设计安全边界 |
| vendor 外部依赖 | ggml + cJSON 锁定 commit 纳入仓库 | 构建系统 + CI | 中——改为系统包管理器需要适配多种发行版 |

### 9.3 新增 / 修改的跨模块契约

```
- libnlc 公共 API (nlc_api.h):
    nl_init(model_path) → int
    nl_parse(input, &result) → int (0=成功)
    nl_free()
    
- nl-perm IPC 协议 (Unix socket):
    REQUEST:  {type, pid, user, cwd, nl_input, generated_cmd, paths[], command_type}
    RESPONSE: {decision: allow|confirm|deny, level: L1|L2|L3|L4, reason, audit_id}
    
- nl-run 退出码:
    0 = 全部成功
    K = 第 K 步失败 (1-indexed)
    -1 = 文件不存在/为空
```
