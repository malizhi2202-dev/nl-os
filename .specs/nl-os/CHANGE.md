## Why（为什么做）

### 现状痛点

| 现有方案 | 本质问题 |
|---|---|
| Shell GPT / Warp / Open Interpreter | 需要**安装 agent + 调云端 API**，不是系统原生能力 |
| Claude Code / Copilot | **代码补全**工具，不是操作系统交互方式 |
| llama.cpp / Ollama | 只是**推理引擎**，没有 OS 集成 |
| Android / iOS AI 助手 | **应用层** AI，受沙箱限制，不能操作系统 |

**核心 gap**：目前没有人在做「把 NL 理解能力变成 OS 的第 5 个基础抽象」（继进程、文件、内存、网络之后）。

### 用户需求映射

用户需求文档 21 条按主题聚类：

```
需求 #1-2    →  实现约束：C 语言、多架构优化、小模型
需求 #3-4    →  权限模型：4 级分级
需求 #5      →  运维：模型更新
需求 #6-7    →  安全：命令映射不能出错
需求 #8-9    →  部署：无需安装、快速启动
需求 #10-13  →  NL-C 编程范式（.tools 脚本 + NL-C 语言）

需求 #21     →  终极愿景：NL-OS

## 重新定义产品：三个独立可交付的子系统

不是"一个大系统分三期"，而是**三个独立产品**，每个都可以单独使用，也可以组合：

```
┌──────────────────────────────────────────────────┐
│                    NL-OS                          │
│                                                   │
│  ┌─────────────┐  ┌─────────────┐  ┌───────────┐ │
│  │  NL-Shell   │  │   NL-C      │  │ 权限系统   │ │
│  │  (终端交互)  │  │  (编程语言)  │  │ (安全底座) │ │
│  │             │  │             │  │           │ │
│  │ 替换 bash   │  │ .tools 脚本 │  │ L1 读安全  │ │
│  │ 自然语言输入 │  │ .nl 源文件  │  │ L2 修改确认│ │
│  │ → 命令执行  │  │ 包管理      │  │ L3 删除高危│ │
│  │             │  │ C 解释器    │  │ L4 禁止执行│ │
│  └──────┬──────┘  └──────┬──────┘  └─────┬─────┘ │
│         │                │               │       │
│  ┌──────┴────────────────┴───────────────┴─────┐ │
│  │          NL 推理引擎（实现层）                │ │
│  │  · C 语言编写 · 纯 CPU · 量化模型 (2B-4B)    │ │
│  │  · 三层混合架构（规则 → 意图分类 → 模板填充） │ │
│  └─────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────┘
```

### 产品 A：NL-Shell —— 自然语言终端

**一句话**：把 bash 的 `命令 参数` 模式升级为自然语言输入。

```
$ 把上周修改过的 python 文件列出来
→ find . -name "*.py" -mtime -7 -type f

$ 把 test 目录压缩成 tar.gz
→ tar -czf test.tar.gz test/

$ 显示占用内存最多的 3 个进程
→ ps aux --sort=-%mem | head -4
```

**关键设计**：
- 不是 bash 的替代品，是 bash 的**自然语言前端**——底层仍然执行标准 Linux 命令
- 可以 pipe 到传统命令：`$ 找到所有大于 100M 的文件 | xargs du -sh | sort -rh`
- 支持混合模式：自然语言 + 传统命令混用
- 误解析时用户可以看生成的命令并拒绝执行（dry-run 模式）

### 产品 B：NL-C —— 自然语言编程语言

**一句话**：让自然语言成为一种可执行、可版本控制、可模块化的编程范式。C 语言是它的"汇编层"。

**两层设计**：

| 层 | 文件名 | 复杂度 | 类比 | 示例 |
|---|---|---|---|---|
| **NL-Script** | `.tools` | 简单 | Bash 脚本 | 一系列 NL 指令，顺序执行 |
| **NL-C** | `.nl` | 完整 | C/Python 源文件 | 带函数/包/导入/类型的 NL 程序 |

**.tools 文件示例**（MVP 即可用）：
```
# setup-dev.tools
创建一个叫 build 的目录
进入 build 目录
运行 cmake ..
运行 make -j4
如果编译成功，把生成的二进制复制到 /usr/local/bin/
```

**.nl 文件示例**（二期）：
```
# data-pipeline.nl
@import file_ops, network_ops

定义一个函数 下载并解压(url: 字符串, 目标目录: 路径) {
    从 {url} 下载文件到 /tmp/download/
    解压到 {目标目录}
    返回 解压后的文件列表
}

对 data_sources.txt 里的每一行调用 下载并解压
```

**包系统设计**（需求 #11-12）：

```
nl_packages/
├── file_ops/       # 文件操作包
│   ├── package.nl  # 包定义 + 导出
│   └── src/        # C 实现
├── network_ops/    # 网络操作包
├── sys_admin/      # 系统管理包
└── user_packages/  # 用户自定义包
```

### 产品 C：权限分级系统 —— 安全底座

**一句话**：NL-OS 的操作权限分级，防止自然语言的模糊性导致不可挽回的破坏。

**4 级权限模型**（直接从需求文档提取）：

| 级别 | 标签 | 触发条件 | 行为 | 示例 |
|---|---|---|---|---|
| **L1** | 读安全 | 读操作 | 自动放行 | `ls`, `cat`, `find`, `ps` |
| **L2** | 修改需确认 | 写/修改操作 | 显示将要执行的操作，用户回车确认 | `touch`, `mkdir`, `echo > file` |
| **L3** | 删除高危 | 删除/不可逆操作 | 红色警告 + 需要用户输入 "yes" | `rm`, `rmdir`, `mv` (覆盖), `dd` |
| **L4** | 禁止执行 | 明确危险 | 硬拦截，**无法通过 NL-Shell 执行** | `rm -rf /`, `chmod 777 /`, 裸 `dd` 写设备 |

**L4 的关键设计**：即使模型解析错误，也不可能执行。L4 不是"提示确认"，而是**根本不给通过**。用户必须退回到原生 bash 才能执行 L4 命令。这就满足了需求 #7："不能让用户误操作，或者是模型解析映射错误达成不可挽回措施"。

## 三个核心决策（v2 修订版）

### 决策 1 · MVP 范围（修正）

**修正前**：D(推理引擎) + A(NL-Shell) + C(权限系统)，把 B(NL-C) 推到二期。

**修正后**：MVP = **NL-Shell + 权限系统 + .tools 脚本**。理由：

- 推理引擎是实现手段，不是产品——它是 NL-Shell 内部的一个模块
- `.tools` 是 NL-C 的 MVP 形态——不需要完整的包系统/导入机制，只需要"自然语言脚本能跑"
- .tools 让用户立刻感受到 NL-C 的价值（"我写了一个 .tools 文件，终端 `run setup.tools` 就能自动搭建环境"）
- 真正的 NL-C 完整编程语言（包系统/多范式/跨平台）放到二期

**MVP 可交付物**：
```
nl-os/
├── bin/
│   ├── nlsh           # NL-Shell 主程序（替代 bash 入口）
│   └── nl-run         # .tools / .nl 文件执行器
├── lib/
│   ├── libnlc.so      # NL 理解库（规则引擎 + 模型推理 + 模板填充）
│   └── libperm.so     # 权限检查库
├── models/
│   └── nl-small.gguf  # 默认量化小模型
├── etc/
│   ├── nlsh.conf      # NL-Shell 配置
│   └── perm_rules.conf # 权限规则配置
└── packages/          # 内置命令包（硬编码，二期变成可扩展）
    ├── file.c         # 文件操作
    ├── dir.c          # 目录操作
    └── proc.c         # 进程操作
```

### 决策 2 · 模型策略（深化）

**维持三层混合架构，但明确各层职责和边界**：

```
用户输入："把 downloads 里大于 100M 的视频删掉"
       │
  ┌────▼──────────────────────────────────────┐
  │  Layer 1: 安全网关（最先执行，不可绕过）      │
  │  · 检测到 "删" → 判定为 L3 高危操作         │
  │  · 提取路径 → 检查黑名单（/ /etc /boot...） │
  │  · 路径在黑名单 → 升级为 L4，硬拦截         │
  │  · 路径安全 → 放行到 Layer 2               │
  └────────────┬─────────────────────────────┘
               │
  ┌────────────▼─────────────────────────────┐
  │  Layer 2: 规则引擎（关键词 + 模板匹配）      │
  │  · 命中模板 "删除 {path} 里 {condition} 的 │
  │    {filetype}" → 填充槽位，跳到 Layer 4   │
  │  · 未命中 → 进入 Layer 3                  │
  └────────────┬─────────────────────────────┘
               │
  ┌────────────▼─────────────────────────────┐
  │  Layer 3: 小模型意图提取（仅做理解）         │
  │  · 输入："把 downloads 里大于 100M 的      │
  │    视频删掉"                               │
  │  · 输出 JSON：                             │
  │    {"intent":"delete",                     │
  │     "target":"video files >100M",          │
  │     "location":"~/downloads"}              │
  └────────────┬─────────────────────────────┘
               │
  ┌────────────▼─────────────────────────────┐
  │  Layer 4: 命令生成器（模板填充，模型不写命令）│
  │  · intent=delete + location + condition   │
  │  · → find ~/downloads -name "*.mp4"       │
  │      -size +100M -delete                  │
  │  · 输出命令给用户 review（L3 必须确认）     │
  └───────────────────────────────────────────┘
```

**关键安全设计**：Layer 1（安全网关）在模型之前运行，不是之后。这样即使模型幻觉，危险路径也被拦在外面。

### 决策 3 · 内核集成路径（深化）

**维持 D→B 渐进，但明确每个阶段的触发条件**：

```
Phase 1（MVP）           Phase 2              Phase 3
用户态权限守护进程   →    eBPF 内核拦截     →   发行版集成
                       · LSM hook (landlock)
· seccomp 约束进程       · 系统调用过滤       · 预装于 Linux 发行版
· PAM 集成              · 无内核模块风险      · 作为 /bin/nlsh 分发
· audit 日志                                           · 与 systemd 集成
· 零内核依赖
```

**为什么 Phase 1 不需要内核态**：
- 权限检查在 NL-Shell 进程内完成——L4 命令根本不会被执行
- 如果用户绕过 NL-Shell 直接用 bash，那是用户自己的选择（就像 `sudo rm -rf /` 一样）
- NL-Shell 的职责是**防止 NL 模糊性导致的误操作**，不是取代 Linux 的权限系统

## MVP 深度定义（第一期交付）

### 功能边界

```
✅ 终端输入自然语言 → 正确 Linux 命令 → 执行 → 显示结果
✅ 覆盖 3 类核心操作：文件（创建/删除/移动/查看）/ 目录（导航/列表）/
   进程（查看/搜索/信号）
✅ 4 级权限生效：L2 确认 / L3 警告 / L4 硬拦截
✅ .tools 文件：编写 → 执行（像 bash 脚本一样）
✅ 规则引擎命中率 > 80%（高频命令零延迟）
✅ 模型推理作为兜底（规则未命中时）
✅ 用户可以随时用 `--dry-run` 查看生成的命令而不执行
✅ 混合模式：自然语言和传统命令可以混用

❌ 不做 .nl 完整编程语言（函数/导入/包管理）
❌ 不做用户自定义包
❌ 不做多用户
❌ 不做 GUI/语音
❌ 不做内核模块
❌ 不做模型微调
```

### 性能目标

| 指标 | 目标 | 测量方式 |
|---|---|---|
| NL-Shell 冷启动 | < 100ms | `time nlsh -c "echo test"` |
| 模型加载 | < 2s（首次）/ < 100ms（热缓存） | 加载 GGUF 到内存 |
| 规则匹配延迟 | < 1ms | 单条命令 |
| 模型推理延迟 | < 500ms | 单次意图提取 |
| 内存占用（空闲） | < 500MB（含模型）/ < 50MB（不含模型） | `/proc/pid/status` VmRSS |
| 二进制大小 | < 5MB（不含模型） | `strip` 后 |

## 验收线（粗粒度）

1. 用户在新终端中输入 `nlsh`，进入 NL-Shell，输入"创建一个叫 test 的文件夹"，系统执行 `mkdir test` 并提示成功
2. 用户输入"删掉根目录"，系统在 Layer 1 安全网关拦截（L4），输出"此操作被禁止，原因：目标路径 / 在黑名单中"，且**不生成任何命令**
3. 用户输入"删除 /tmp/test 目录"，系统判定为 L3，显示将执行的命令 `rm -rf /tmp/test`，要求用户输入 `yes` 确认
4. 用户创建 `build.tools` 文件，写入 4 条 NL 指令，执行 `nl-run build.tools`，4 条指令依次执行并输出结果
5. 规则引擎在 100 条预设高频命令的测试集上，命中率 > 80%
6. 在树莓派 4（4GB RAM）上，NL-Shell 冷启动 < 100ms，模型推理延迟 < 500ms
7. 用户在 NL-Shell 中直接输入 `ls -la | grep nl`，传统命令正常透传执行

## 风险与未知

| 风险 | 等级 | 缓解措施 |
|---|---|---|
| 2B 模型中文 NL 理解准确率不足 | 🔴 高 | 小模型只做意图提取（分类问题比生成问题容易）；规则引擎覆盖 80% 高频场景；建立评测基准后再选模型 |
| 模型幻觉导致危险命令映射 | 🔴 高 | 安全网关前置（Layer 1 在模型前拦截危险路径）；模板填充限制命令结构；L4 黑名单硬编码 |
| CPU 推理延迟影响交互体验 | 🟡 中 | 规则引擎覆盖高频→大部分请求零推理延迟；量化到 Q4_0 减少计算量；可选更小模型（1.5B） |
| 规则太多退化为传统 shell | 🟡 中 | 通过真实用户数据驱动规则覆盖范围；规则只覆盖"确定性高"的场景，模糊场景留给模型 |
| NL-C 语法设计缺乏参考 | 🟡 中 | .tools 先跑通简单场景（指令序列），积累使用数据后再设计 .nl 复杂语法 |
| 社区接受度（vs 传统 shell）| 🟢 低 | 兼容传统命令透传；关注开发者和运维人员体验；开源社区驱动 |

## 对抗性审查修正（架构师 vs 审计师）

经过 Builder vs Skeptic 四轮对抗，以下修正已纳入方案：

| # | 发现 | 严重度 | 修正 |
|---|---|---|---|
| 1 | 路径黑名单可被 `../` `symlink` 绕过 | 🔴 严重 | Layer 1 增加 `realpath()` 规范化 + **白名单**（仅 `/home /tmp /var /opt` 允许写操作），黑名单是最后防线 |
| 2 | 模板填充的槽位（target）来自模型输出，仍可能危险 | 🔴 严重 | 模型输出的实体（路径/文件名）必须通过白名单 + `stat()` 验证，不通过则拒绝执行并提示用户重新描述 |
| 3 | 脚本自动化（`.tools`）无权限保护 | 🟡 中等 | `.tools` 支持头部声明 `# @perm: allow-l2` 预授权；`--batch` 模式生成命令后需人工审核 |
| 4 | 2B 模型在 ARM64 的性能/内存缺乏基准 | 🟡 中等 | **先建评测基准再选模型**：设计 1000 条中文 NL→命令测试集，对比 0.5B/1.5B/2B 三类模型的准确率+延迟+内存；意图分类可能 0.5B 就够了 |
| 5 | 模型冷加载 2s + 推理 500ms = 首次体验差 | 🟡 中等 | 模型 mmap 常驻（规则优先，模型在规则未命中时才用到）；意图+实体提取合并为单次结构化推理 |
| 6 | `.tools` 的非确定性——同一句话今天和明天生成不同命令 | 🟡 中等 | 支持 `nl-run --freeze`：首次执行时保存具体命令，后续直接执行已保存命令（不经过模型），类似编译 |
| 7 | 模型可能需要多轮推理（意图→实体→消歧） | 🟢 低 | 单次结构化输出（JSON Schema 约束），一次推理同时输出 intent + entities + confidence |

### 对抗后确认的核心风险（无法完全消除，需持续监控）

1. **对抗性输入不可穷举**：NL 理解本质是统计模型，无法像 seccomp 那样数学上证明安全。缓解：安全网关不依赖模型（硬编码规则）+ L4 黑名单持续更新 + 社区驱动的对抗样本库
2. **模型更新引入回归**：更新模型版本可能导致之前正确的 NL 映射变错。缓解：建立回归测试集（每次模型更新前跑）+ 支持模型版本锁定
3. **"模型理解错误 + 脚本自动化 + 权限盲区"三明治攻击**：这是最难防的组合。缓解：.tools 默认运行在 `--dry-run` 模式（只输出不执行），用户需显式加 `--execute` 才真正执行

### 第一场：代码架构与模块边界

**🟫 架构师（开火）**：
当前方案说 MVP 三个模块（NL-Shell / 推理引擎 / 权限守护），但模块边界太模糊。我要求回答：

1. **推理引擎是库还是进程**？如果是 `libnlc.so`，那 NL-Shell link 它，权限守护也 link 它？两个进程各自加载一份模型到内存？在 4GB 的树莓派上你敢这么干？

**🟦 研发负责人（回应）**：

架构师说得对。我的方案：

```
nl-os/
├── src/
│   ├── libnlc/           # 核心库（推理 + 规则引擎）
│   │   ├── rule_engine.c  # 规则匹配（零依赖，纯 C99）
│   │   ├── model_loader.c # GGUF 加载器（依赖 ggml）

│   │   ├── template_fill.c  # 模板填充
│   │   └── path_safety.c    # 路径规范化 + 白名单
│   │
│   ├── nlsh/             # NL-Shell 可执行文件
│   │   ├── main.c         # REPL 入口
│   │   ├── line_editor.c  # readline 替代（免 GPL 污染）
│   │   ├── dispatch.c     # 传统命令透传 / NL 路由
│   │   └── history.c      # 命令历史
│   │

│   │   ├── ipc.c          # Unix domain socket 通信
│   │   └── audit_log.c    # syslog 审计
│   │
│   └── nl-run/           # .tools 执行器
│       ├── main.c
│       └── script_parser.c
│
├── tests/

**关键架构决策**：
- `libnlc` 作为**共享库**，NL-Shell 和权限守护通过 `dlopen` 加载，**模型只加载一次**（共享内存 mmap）
- 权限守护用 **Unix domain socket** 通信，NL-Shell 发操作请求 → 权限守护返回 `allow/deny/confirm` → NL-Shell 执行或拒绝
- C11 标准（ARM64 和 x86_64 的 GCC/Clang 都完整支持，C17 在旧 ARM 交叉编译链上可能有坑）

**🟫 架构师（追问）**：
权限守护如果进程挂了怎么办？NL-Shell 是等它重启还是直接放行？这是安全关键路径——"挂了就放行"= 安全机制形同虚设；"挂了就拒绝所有操作"= 系统不可用。

**🟦 研发负责人**：
**fail-closed 原则**——权限守护不可达时，NL-Shell **拒绝所有 NL 发起的操作**，但保留传统命令透传（`ls -la` 这种直接透传的不过权限守护）。同时：
- 权限守护由 systemd 管理，崩溃自动重启（`Restart=always`）
- NL-Shell 启动时检查权限守护存活，死了就报错退出（`nlsh: permission daemon not running, aborting`）
- 权限守护内部用 seccomp 限制自己的系统调用（防止被攻陷后扩大危害）

### 第四场：Agent 框架设计

**🟨 Agent 开发（主攻）**：
当前方案把 NL-Shell 定位为"单次指令翻译器"——用户说一句话，系统翻译成命令，执行，结束。但需求文档里有非常关键的一条你们忽略了：

> "用户输入进入 xxx 项目可以自己扫描出目录供用户选择进入"

这**不是**单次指令翻译。这是**多步交互式 Agent**：

```
用户："进入 my-project 项目"
      │
  ┌───▼────────────────────────────────────┐
  │ Step 1: 理解意图 "enter_project"        │
  │ Step 2: find ~/ -name "my-project"     │  ← 调用 find 命令
  │ Step 3: 如果有多个匹配 → 列出供用户选择  │  ← 交互式确认
  │ Step 4: cd 到选中的目录                  │  ← 执行操作
  │ Step 5: 可选 → 自动运行 git status     │  ← 上下文感知的后续动作
  └────────────────────────────────────────┘
```

三步以上的操作，需要一个 **Agent 循环**：

```
┌─────────────────────────────────────────────┐
│              NL-OS Agent Loop                │
│                                              │
│  用户输入                                     │
│     ↓                                        │
│  ┌──────────────────────┐                    │
│  │  Planner（规划）       │ ← 大模型/规则     │
│  │  拆解为步骤序列        │                    │
│  └─────────┬────────────┘                    │
│            ↓                                  │
│  ┌──────────────────────┐                    │
│  │  Executor（执行）      │ ← 调用系统命令    │
│  │  逐步执行，收集结果    │                    │
│  └─────────┬────────────┘                    │
│            ↓                                  │
│  ┌──────────────────────┐                    │
│  │  Observer（观察）      │ ← 解析命令输出    │
│  │  判断是否达到目标      │                    │
│  └─────────┬────────────┘                    │
│            ↓                                  │
│     ┌─── 完成？───┬─── 否 ──→ 回到 Planner   │
│     │             │                          │
│    是           失败                          │
│     ↓             ↓                          │
│  返回结果      向用户求助                     │
└─────────────────────────────────────────────┘
```

**🟫 架构师（挑战）**：
这个 Agent Loop 放在哪里？NL-Shell 里？那权限守护怎么介入每一步？如果 Agent 规划了 5 步，第 3 步是 `rm`，但用户在第一步后就离开了，谁来中止？

**🟨 Agent 开发（回应）**：
Agent Loop 在 NL-Shell 内，但**每一步执行前都必须经过权限守护**。架构应该是：

```
NL-Shell (Agent Loop)
  │
  ├─ Planner ──→ 输出步骤序列 [{step1}, {step2}, ...]
  │                  │
  └─ Executor ←─────┘
       │
       ├── 对每一步：
       │   1. 生成具体命令
       │   2. → 权限守护审核 ← (每一步都审！)
       │   3. 用户确认（如果权限守护要求）
       │   4. 执行命令
       │   5. 收集输出，交给 Observer
       │
       └── Observer 判断是否继续
```

**关键安全设计**：
- Agent 生成的**步骤序列**在首次展示给用户（"我将执行以下 3 步：1. xxx 2. xxx 3. xxx，是否继续？"）
- 如果用户离开，Agent 超时自动中止
- 每一步执行前后都写 audit log

**🟦 研发负责人（担忧）**：
Agent Loop 增加了巨大的复杂度。MVP 阶段真的要这个吗？我建议分两级：

| 模式 | MVP | 二期 |
|---|---|---|
| **单步模式** (当前方案) | ✅ NL 输入 → 一条命令 → 执行 | — |
| **Agent 模式** | ⚠️ 只支持 `.tools` 脚本（预定义的步骤序列，无动态规划） | ✅ 完整 Agent Loop（动态规划 + 多步推理） |

MVP 的 `.tools` 就是 Agent 的 MVP 形态——步骤是**人写的**（不是 AI 规划的），但执行是自动化的。这让 Agent 框架的风险可控：AI 只做单步翻译，不做多步规划。

**🟨 Agent 开发（妥协）**：
接受。但 `libnlc` 的 API 设计要**预留 Agent 扩展点**：
- `nl_plan(input) → step_sequence`（目前返回单步，但接口设计为返回序列）
- `nl_execute(step) → result`（单步执行）
- `nl_observe(results) → status`（目前只返回成功/失败，将来返回 `{done: false, next_action: ...}`）

如果 API 一开始不预留，二期重构成本巨大。

**🟫 架构师（拍板）**：
同意。`libnlc` 的公共 API 设计为面向 Agent 的接口，MVP 实现单步退化版本。这是"接口先行，实现渐进"。

### 第五场：测试策略

**🟦 研发负责人（主攻）**：
当前方案的测试策略太粗糙。对一个"NL→命令→执行"的系统，测试分五层：

```
Layer 5: 端到端测试（真实 NL 输入 → 真实命令执行）
         └── Docker 容器隔离，每个测试用例独立 filesystem snapshot
Layer 4: 集成测试（libnlc + nl-perm + nlsh 联调）
         └── 模拟终端输入，捕获命令输出
Layer 3: 模型推理测试（给定 prompt → 验证输出 JSON schema）
         └── 离线跑，不依赖真实文件系统
Layer 2: 组件测试（规则引擎/模板填充/路径安全 各自独立测）
         └── 纯 C 函数测试，无 IO，无模型
Layer 1: 单元测试（每个 .c 的公开函数）
         └── 编译期跑，CI 必过
```

**测试金字塔倒过来才是安全的**——Layer 1-3 跑得快（秒级），在 CI 每次 commit 都跑。Layer 4-5 跑得慢（分钟级），在 PR merge 前跑。

**🟩 资产开发（补充）**：
模型层的测试（Layer 3）和其他测试不一样——模型的结果不是二元的 pass/fail，而是**准确率分布**。测试策略：

1. **回归测试**（每次改 prompt 模板时）：500 条固定测试集，准确率不能下降 > 2%
2. **对抗性测试**（每次发布前）：500 条对抗样本，误操作率必须 = 0（安全相关）
3. **A/B 测试**（模型选型时）：同一测试集跑多个模型，输出对比报告

**🟨 Agent 开发（补充）**：
`.tools` 脚本的测试也要考虑**幂等性**——同一个 `.tools` 跑两次，结果应该一致。测试方法是：Docker 里跑一遍 `.tools`，snapshot 文件系统，恢复 snapshot，再跑一遍，diff 比较。
