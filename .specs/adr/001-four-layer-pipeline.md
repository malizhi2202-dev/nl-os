# ADR-001: 四层混合管线架构

- **状态**: accepted
- **日期**: 2026-07-01
- **关联决策**: D1, D6, D9

## Context

NL-Shell 需要将自然语言输入转换为 Linux 命令。面临核心矛盾：纯规则引擎（如关键词匹配）无法覆盖自然语言的多样性；纯模型生成（端到端 seq2seq）存在幻觉风险，可能生成危险命令。需求文档明确要求"不能让用户误操作，或者是模型解析映射错误达成不可挽回措施"。

## Decision

采用四层混合管线，按顺序执行：

1. **Layer 1: 安全网关** — 在模型之前运行。剥离 Unicode 控制字符 → 正则提取路径 → `realpath()` 规范化 → 白名单+黑名单校验。涉及 L4 路径的直接拒绝，不生成命令。
2. **Layer 2: 规则引擎** — 关键词 + 模板匹配。覆盖 80% 高频命令，O(1) 查找，< 1ms 延迟。
3. **Layer 3: 模型推理** — 仅做意图分类 + 实体提取（结构化 JSON 输出），不做命令生成。confidence < 0.6 拒绝执行。
4. **Layer 4: 模板填充** — 根据 intent + entities 填充预定义命令模板。模型不直接生成命令文本。

**关键设计原则**：安全网关是硬编码的（不依赖模型）、规则引擎是确定性的（同输入→同输出）、模型只做分类（比生成可靠）、模板限制命令结构（防止注入）。

## Consequences

- ✅ 安全网关前置：即使模型幻觉，危险路径也被拦截
- ✅ 规则引擎保证 80% 场景零延迟 + 100% 确定
- ✅ 模型只做分类（准确率高于生成），小模型可用
- ❌ 表达能力受限于预定义 intent 枚举和模板数量
- ❌ 四层管线增加代码复杂度，层间接口需严格定义
- ❌ 规则和模型的边界划分是持续工程挑战（数据驱动优化）
