# ADR-003: NL-C 两层语法设计 (.tools → .nl)

- **状态**: accepted
- **日期**: 2026-07-01
- **关联决策**: D8

## Context

需求文档描述了两种 NL 编程形态：`.tools` 脚本（自然语言指令序列，类比 bash 脚本）和 NL-C 语言（C 作为自然语言解释器，含包系统/导入/多范式）。如果直接从完整的 NL-C 语言开始，可能陷入过早抽象的陷阱——在还不知道用户怎么用 NL 编程的情况下，就设计复杂的包系统和类型系统。

## Decision

**分两层交付**：

**Phase 1 (MVP): `.tools` 脚本**
- 纯自然语言指令序列，一行一条 NL 指令
- `nl-run <file.tools>` 逐行执行
- 支持头部声明：`# @perm: allow-l2`（预授权）
- 支持 `--freeze`（首次执行后锁定为具体命令）
- fail-fast：任何步骤失败立即停止

**Phase 2: `.nl` 源文件**
- 函数定义、包导入、类型注解
- 完整的模块系统（无循环引用检测）
- 多范式支持（指令式/函数式）
- 语法规范基于 .tools 的经验数据设计

**MVP 不做 .nl**。先用 .tools 积累用户数据（什么场景下用户写 NL 脚本？多复杂的脚本？），再基于数据设计 .nl 语法。

## Consequences

- ✅ `.tools` 是 MVP 可验证的最小 NL 编程单元，降低第一期复杂度
- ✅ 积累真实用户数据后再设计 .nl 语法，避免闭门造车
- ✅ `.tools` 本身就有独立价值（替代简单 bash 脚本）
- ❌ .tools 的简单性可能限制早期用户场景（无法做条件分支/循环）
- ❌ .tools → .nl 的迁移路径需要在 .nl 设计时考虑兼容性
